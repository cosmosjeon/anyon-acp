# UX Design Validation Checklist

## 📋 문서 구조

- [ ] YAML frontmatter가 완전함 (service_type, platform, screens, flows 포함)
- [ ] PRD 참조가 명시됨 (based_on_prd)
- [ ] 모든 섹션이 존재하고 placeholder가 없음
- [ ] 총 화면 수와 플로우 수가 메타데이터에 기록됨

## 🔗 PRD 일관성

- [ ] PRD의 service_type과 일치함
- [ ] PRD의 platform과 일치함
- [ ] PRD의 모든 core_features가 UX에 반영됨
- [ ] PRD의 target_users에 맞는 UX 디자인임

## 🎯 UX 설계 원칙

- [ ] 플랫폼에 적합한 UX 원칙이 정의됨
- [ ] 타겟 사용자에 맞는 원칙이 포함됨
- [ ] 각 원칙에 구체적인 적용 예시가 있음
- [ ] 3-6개의 명확한 원칙이 있음

## 📱 화면 구조

- [ ] 모든 필요한 화면이 식별됨 (빠진 화면 없음)
- [ ] 각 화면의 목적이 명확히 정의됨
- [ ] 각 화면의 진입점/이탈점이 명시됨
- [ ] 인증 화면들이 포함됨 (로그인, 회원가입 등)
- [ ] 시스템 화면들이 포함됨 (설정, 프로필 등)
- [ ] 에러 화면들이 포함됨
- [ ] Empty state 화면들이 고려됨
- [ ] 로딩 상태가 정의됨

## 📊 화면 계층 구조

- [ ] 네비게이션 패턴이 정의됨
- [ ] 화면 계층이 논리적으로 구성됨
- [ ] Top-level 화면이 3-5개로 적절함 (cognitive load)
- [ ] 모든 화면이 도달 가능함 (orphan 없음)
- [ ] Parent-child 관계가 명확함
- [ ] Modal/overlay 화면이 구분됨

## 🔥 사용자 플로우 (가장 중요!)

### 플로우 구성
- [ ] 표준 플로우와 커스텀 플로우가 명확히 구분됨
- [ ] 총 플로우 수 = 표준 플로우 + 커스텀 플로우
- [ ] 플로우 통계가 문서에 기록됨

### 표준 플로우 (자동 생성)
- [ ] PRD와 platform 분석을 통해 필요한 표준 플로우가 식별됨
- [ ] 인증 플로우가 포함됨 (로그인, 회원가입, 비번 찾기 등)
- [ ] 시스템 화면 플로우가 포함됨 (설정, 프로필 등)
- [ ] 에러/빈 화면 플로우가 포함됨
- [ ] PRD 기능에 맞는 조건부 표준 플로우가 포함됨
  - [ ] 검색 기능 → 검색 플로우
  - [ ] CRUD 기능 → 생성/수정/삭제 플로우
  - [ ] 소셜 기능 → 팔로우/좋아요/댓글 플로우
  - [ ] 결제 기능 → 장바구니/결제 플로우
- [ ] 각 표준 플로우가 platform에 맞게 생성됨 (모바일/웹)
- [ ] 표준 플로우도 병적으로 상세함 (자동이지만 품질 유지)

### 커스텀 플로우 (서비스 고유)
- [ ] PRD의 핵심 기능 중 표준으로 커버 안 되는 것들이 식별됨
- [ ] 서비스 차별화 기능에 대한 플로우가 있음
- [ ] 복잡한 비즈니스 로직이 플로우로 정의됨
- [ ] 각 커스텀 플로우가 왜 커스텀인지 명확함

### 플로우 완전성
- [ ] PRD의 모든 핵심 기능에 대한 플로우가 있음 (표준 + 커스텀)
- [ ] 온보딩 플로우가 정의됨
- [ ] 각 핵심 기능의 complete flow가 있음
- [ ] 총 플로우 수가 적절함 (최소 8-10개 이상 권장)

### 플로우 상세도 (병적으로 구체적이어야 함!)
- [ ] 각 플로우가 단계별로 나뉘어 있음
- [ ] 각 단계마다 화면 상태가 명시됨
- [ ] 각 단계마다 사용자 액션이 구체적으로 정의됨
- [ ] 각 단계마다 시스템 반응이 상세히 기술됨
- [ ] 로딩 상태와 애니메이션이 명시됨
- [ ] 화면 전환 방식이 정의됨 (애니메이션 포함)

### Happy Path (정상 경로)
- [ ] 메인 경로가 시작부터 끝까지 완전함
- [ ] 각 단계가 빠짐없이 정의됨
- [ ] UI 요소와 인터랙션이 구체적으로 명시됨
- [ ] 데이터 변경 사항이 명확함

### Alternative Paths (대안 경로)
- [ ] 같은 목표를 달성하는 다른 방법들이 정의됨
- [ ] 각 대안 경로가 상세히 기술됨
- [ ] 메인 경로와의 합류점이 명확함
- [ ] 언제 대안 경로를 사용하는지 명시됨

### Error Cases (에러 케이스) ⚠️
- [ ] **입력 에러**가 모두 정의됨
  - [ ] 잘못된 형식 입력
  - [ ] 필수 항목 누락
  - [ ] 너무 긴 텍스트
  - [ ] 유효하지 않은 값
- [ ] **네트워크 에러**가 모두 정의됨
  - [ ] 인터넷 끊김
  - [ ] 서버 무응답
  - [ ] 타임아웃
- [ ] **권한 에러**가 정의됨
  - [ ] 카메라/위치 등 권한 거부
  - [ ] 인증 필요한 기능 접근
- [ ] **데이터 에러**가 정의됨
  - [ ] 데이터 없음
  - [ ] 로딩 실패
  - [ ] 중복 데이터
- [ ] **각 에러마다**:
  - [ ] 발생 조건이 명확함
  - [ ] 에러 메시지가 정확한 문구로 작성됨
  - [ ] 메시지 위치와 스타일이 정의됨
  - [ ] 복구 방법이 제시됨
  - [ ] 복구 후 상태가 명확함

### Flow Completeness (플로우 완결성)
- [ ] 종료 조건이 명확함 (성공/실패)
- [ ] 성공 시 피드백이 정의됨
- [ ] 실패 시 복귀 경로가 정의됨
- [ ] 중간 취소 시 동작이 정의됨
- [ ] 뒤로가기 동작이 정의됨
- [ ] 데이터 보존 여부가 명시됨

### Visual Flow Representation
- [ ] 각 플로우에 다이어그램 또는 구조화된 표현이 있음
- [ ] 메인/대안/에러 경로가 시각적으로 구분됨

## 🎨 인터랙션 패턴

### 기본 인터랙션
- [ ] 플랫폼에 적합한 인터랙션이 선택됨
- [ ] Primary 인터랙션이 정의됨 (탭/클릭 등)
- [ ] Secondary 인터랙션이 정의됨 (스와이프/호버 등)
- [ ] 각 인터랙션의 시각적 피드백이 정의됨

### 제스처 (해당 시)
- [ ] 사용되는 모든 제스처가 나열됨
- [ ] 각 제스처가 어디서 사용되는지 명시됨
- [ ] 제스처 동작이 구체적으로 정의됨
- [ ] 제스처 피드백이 정의됨

### 피드백 및 애니메이션
- [ ] 버튼 탭/클릭 피드백이 정의됨
- [ ] 폼 제출 피드백이 정의됨
- [ ] 로딩 표시 방식이 정의됨
- [ ] 성공/에러 피드백이 정의됨
- [ ] 화면 전환 애니메이션이 정의됨
- [ ] 햅틱 피드백이 정의됨 (모바일 해당시)

## 🧩 주요 컴포넌트

- [ ] 재사용되는 UI 컴포넌트가 식별됨
- [ ] 각 컴포넌트의 목적이 명확함
- [ ] 각 컴포넌트의 구조가 정의됨
- [ ] 각 컴포넌트의 상태들이 정의됨 (default, hover, disabled 등)
- [ ] 각 컴포넌트의 인터랙션이 명시됨
- [ ] 컴포넌트 사용 위치가 나열됨

## 📊 상태 관리

- [ ] 인증 상태가 정의됨
- [ ] UI 상태가 정의됨 (theme, active tab 등)
- [ ] 데이터 상태가 정의됨 (loading, loaded, error 등)
- [ ] 네트워크 상태가 정의됨
- [ ] 폼 상태가 정의됨 (해당시)
- [ ] 각 상태의 초기값이 명시됨
- [ ] 상태 변경 조건이 정의됨
- [ ] 상태가 영향을 미치는 UI가 명시됨
- [ ] 상태 지속성이 정의됨 (세션/영구)

## ⚡ 엣지 케이스

### Empty States
- [ ] 첫 사용자 (데이터 없음) 상태가 정의됨
- [ ] 검색 결과 없음 상태가 정의됨
- [ ] 삭제된 콘텐츠 상태가 정의됨

### Offline Scenarios
- [ ] 연결 끊김 시 동작이 정의됨
- [ ] 처음부터 오프라인 시 동작이 정의됨
- [ ] 간헐적 연결 시 동작이 정의됨

### Permission Issues
- [ ] 권한 거부 시 대응이 정의됨 (카메라, 위치 등)
- [ ] 권한 없이 작동 가능한 대안이 있음

### Device Limitations
- [ ] 저장공간 부족 시 대응이 정의됨
- [ ] 저전력 모드 고려됨 (해당시)

### Concurrent Actions
- [ ] 동시 작업 충돌 처리가 정의됨
- [ ] 백그라운드/포그라운드 전환 동작이 정의됨

## 📐 레이아웃 및 그리드

- [ ] 그리드 시스템이 정의됨 (웹의 경우)
- [ ] 반응형 breakpoint가 정의됨 (해당시)
- [ ] 스페이싱 시스템이 정의됨
- [ ] Safe area가 고려됨 (모바일 해당시)
- [ ] 화면 방향 지원이 명시됨 (portrait/landscape)

## 🔔 알림 및 권한

- [ ] 필요한 모든 권한이 나열됨
- [ ] 각 권한의 요청 시점이 정의됨
- [ ] 권한 요청 이유 설명이 포함됨
- [ ] 권한 거부 시 대응이 정의됨
- [ ] 알림 트리거 조건이 정의됨 (해당시)
- [ ] 알림 내용과 딥링크가 정의됨 (해당시)

## ♿ 접근성

- [ ] 키보드 네비게이션이 고려됨 (웹)
- [ ] 스크린 리더 지원이 고려됨
- [ ] 터치 영역 크기가 적절함 (44x44px 이상, 모바일)
- [ ] 색상 대비가 고려됨
- [ ] Alt text 전략이 정의됨

## 📱 반응형 디자인 (해당시)

- [ ] 각 breakpoint별 레이아웃이 정의됨
- [ ] 모바일 레이아웃이 정의됨
- [ ] 태블릿 레이아웃이 정의됨 (해당시)
- [ ] 데스크톱 레이아웃이 정의됨 (해당시)
- [ ] 각 breakpoint에서 변경되는 사항이 명시됨

## 📄 화면별 상세

- [ ] 모든 화면에 대한 상세 spec이 있음
- [ ] 각 화면마다 다음이 정의됨:
  - [ ] 목적
  - [ ] 레이아웃 구조
  - [ ] UI 요소 목록과 설명
  - [ ] 가능한 인터랙션
  - [ ] 상태들 (loading, empty, error, success)
  - [ ] 진입 플로우
  - [ ] 이탈 플로우
  - [ ] 필요한 데이터

## 🔄 문서 일관성

- [ ] 화면 이름이 전체 문서에서 일관됨
- [ ] 플로우에서 언급된 모든 화면이 screen inventory에 있음
- [ ] Screen hierarchy가 실제 화면 목록과 일치함
- [ ] 컴포넌트가 화면 정의와 일치함
- [ ] 상태 관리가 플로우와 일치함

## ✅ AI 가독성

- [ ] 모든 정의가 구체적이고 명확함 (애매모호함 없음)
- [ ] 기술 용어가 일관되게 사용됨
- [ ] 각 플로우가 AI가 구현할 수 있을 만큼 상세함
- [ ] YAML 메타데이터로 빠른 검색 가능함
- [ ] 다음 워크플로우(UI)에 필요한 모든 정보 포함됨

## 🚦 다음 단계 준비

- [ ] UI Design Guide에 필요한 모든 컴포넌트가 정의됨
- [ ] 인터랙션 패턴이 명확해서 UI 라이브러리 선택 가능함
- [ ] 화면 구조가 명확해서 디자인 시스템 적용 가능함
- [ ] 문서가 {default_output_file}에 저장됨

## 🔥 최종 품질 검증

### Obsessive Detail Test (병적인 상세함 테스트)
- [ ] 개발자가 이 문서만 보고 화면을 만들 수 있는가?
- [ ] 모든 버튼 클릭이 정확히 무슨 일을 하는지 알 수 있는가?
- [ ] 모든 에러 상황에서 무슨 메시지가 어디에 나오는지 알 수 있는가?
- [ ] 모든 로딩/애니메이션이 어떻게 보이는지 설명되었는가?

### Completeness Test (완전성 테스트)
- [ ] 사용자가 할 수 있는 모든 행동이 커버되었는가?
- [ ] 빠진 화면이 없는가?
- [ ] 빠진 플로우가 없는가?
- [ ] 빠진 에러 케이스가 없는가?

### Consistency Test (일관성 테스트)
- [ ] PRD와 모순이 없는가?
- [ ] 내부적으로 모순이 없는가?
- [ ] 용어 사용이 일관되는가?

2. ✅ 병적으로 구체적인 사용자 플로우 완성
3. ✅ UI Design Guide 워크플로우로 넘어갈 준비 완료
4. ✅ startup-ui 워크플로우 자동 시작
